# 云发单业务总览 v1

更新时间：2026-04-24  
整理来源：飞书知识库导出内容（云发单技术组、云发单研发组、云发单运营组、研发部门、客服知识库）

## 1. 这套业务在做什么

云发单本质上是一套面向私域卖货场景的“微信机器人自动化发单系统”。

核心思路是：

- 托管个人微信机器人或企业微信账号
- 自动收集消息、订阅素材、处理群消息
- 对淘宝、京东、拼多多等平台商品链接做转链
- 将处理后的消息自动发送到微信群、朋友圈或其他私域场景
- 跟踪订单、统计来源与效果，支撑运营与客服

从业务定位看，它不是单一 App，而是一整套“机器人 + 转链 + 消息分发 + 订单追踪 + 后台管理 + 上游渠道接入”的系统组合。

## 2. 核心业务对象

### 2.1 用户侧产品

- 悦拜 APP：用户前台入口，用户在这里完成登录、扫码、验证码、人脸验证、续费、试用、设备位管理等操作。
- 云发单：围绕机器人发消息、跟群、跟圈、源头订阅、素材分发、转链等能力展开的主业务。
- 客服知识库中还出现了鲤享（短剧）APP、美天赚等内容，但当前归档群问题的主线仍然是云发单/悦拜/机器人体系。

### 2.2 机器人与终端

- 微信机器人：通过 iPad 协议、PC Hook、Mac 协议等方式托管个人微信。
- 企业微信/企微：部分链路通过企微处理群消息、发送图片、跟发等。
- 个微：个人微信生态，常与企微互通、视频号同步、素材下载等问题相关。

### 2.3 上游厂商、端别、登录设备

文档里反复出现的核心上游包括：

- 易可云
- 探鲸
- 发单帝
- 云密码
- 小邪

这些上游不是单一“线路名”，而是上游能力提供方。  
在上游下面，还要继续区分两层：

- 端别：微信端、企微端
- 登录设备：Mac、iPad、AndroidPad、Windows、PC 等

也就是说，业务上更接近下面这种结构：

- 发单帝 -> 微信端 -> Mac
- 探鲸 -> 微信端 -> iPad / AndroidPad
- 探鲸 -> 企微端 -> 企微 iPad
- 小邪 -> 微信端 -> iPad / Mac / AndroidPad
- 小邪 -> 企微端
- 云密码 -> 微信端 -> Windows

所以后面做 Case 边界判断时，不能把“上游”“端别”“登录设备”混成一层。

### 2.4 后台与管理端

- office 管理后台：经常用于查看设备、群聊、消息、队列、账号状态、回调日志、用户列表等。
- 商家后台 / 上游后台：尤其是探鲸、易可云等自有后台。
- 云发单 API / office 接口 / 机器人管理端：属于技术与后台能力层。

### 2.5 业务域

从文档看，云发单至少包含这些高频业务域：

- 登录与换号
- 群消息接收与同步
- 跟群 / 跟圈 / 订阅 / 群关注
- 转链
- 订单追踪 / 订单异常 / 统计规则
- 设备位、有效期、续费、试用
- 页面/H5/域名/DNS
- 图片、视频号、媒体资源

## 3. 典型系统边界

后面做 Case 合并判断时，最重要的不是“都叫登录不上”，而是“到底是哪一层登录不上”。

### 3.1 悦拜 / 云发单前台

特征词：

- 悦拜 APP
- 云发单首页
- 续费
- 试用
- 设备位
- 首页空白
- H5 页面
- 页面打不开
- DNS

这类问题更偏用户前台、H5、账户状态、续费试用、前端展示。

### 3.2 云发单机器人能力层

特征词：

- 跟群
- 跟圈
- 激活群
- 素材群
- 消息接收
- 群消息丢失
- 不跟发
- 同步历史消息

这类问题更偏机器人消息链路与业务逻辑。

### 3.3 上游厂商/端别/设备层

特征词：

- 探鲸
- 易可云

- 云密码

- 企微
- 个微
- 微信端
- iPad
- Mac

这类问题往往不是“云发单统一问题”，而是具体上游、具体端别、具体登录设备的问题。

### 3.4 转链与联盟能力层

特征词：

### 3.5 订单与统计层

特征词：

- 订单
- 出单
- 跟单规则
- 统计
- 爆款商品
- 渠道管理 id
- 推广位 id

这类问题是订单/统计域，不应和登录/消息问题混并。

## 4. 重要术语

以下术语会反复出现在 case 中，建议作为模型词典：

- 渠道管理 id：平台用户授权给我方后的用户身份识别 id。
- 推广位 id：用于追踪商品推广入口的标识。
- 会场 id：淘宝/京东/拼多多等活动页编号。
- 动态 id 转链：为追踪链路改造后的商品 id 场景。
- 消费者比价场景：用户自购/分享前的价格比较场景。
- 联盟口令：来自淘宝联盟、京东联盟、拼多多联盟等后台的推广口令。
- 主站口令：直接从平台 App 或 PC 复制的原始口令/链接。
- 设备位：用户在云发单中持有的机器人/终端席位，和登录稳定性、有效期、续费直接相关。
- 群消息接收：部分上游（尤其探鲸）必须先开启群消息接收，才会把群消息回传给我方。
- 激活群 / 素材群 / 订阅：云发单里不同的消息分发与跟发场景。

## 6. 对 Case 归档最有帮助的结论

### 6.1 “同一个症状”不等于“同一个问题”

例如：

- “登录不上云发单前台”
- “探鲸线路登录不上”
- “企微扫码后验证码验证失败”

都叫“登录问题”，但它们的业务对象、责任边界、排查路径都不同。

### 6.2 优先按“业务对象”而不是“动词”拆问题

例如：

- “后台看不到订单” 是订单/后台问题
- “用户收不到验证码” 是登录/验证码链路问题
- “跟群不生效” 是机器人/群消息问题
- “淘宝链接转不了” 是转链问题

### 6.3 上游名称是强边界，但还不够

如果消息里明确出现：

- 探鲸
- 易可云
- 发单帝
- 云密码
- 小邪

默认先把它当成不同系统边界，而不是简单并入“登录问题”或“消息问题”。

但在同一个上游内部，还要继续细分：

- 是微信端还是企微端
- 使用的登录设备是什么
- 出问题的是登录、收消息、发图片、跟群、视频号还是别的能力

例如：

- “探鲸企微端发图失败”
- “探鲸微信端 iPad 登录失败”

虽然都属于探鲸，但也不能直接视为同一个问题。

### 6.4 前台问题与后台问题要分开

例如：

- “APP 页面空白”
- “office 后台没数据”

就算都发生在同一时间，也不应默认合并。

## 7. 这份文档适合怎么用

推荐用途：

- 作为 Case 合并/新建判定模型的 system prompt 背景材料
- 作为客服/研发/产品对齐业务对象的速查说明
- 作为后续知识检索系统的顶层目录说明

不推荐直接拿它回答具体用户问题；具体问题解答仍应回到原始知识文档检索。

# Case 边界判定说明 v1

更新时间：2026-04-24  
用途：给“是否合并到已有 Case”这一步提供业务边界  
适用对象：归档模型、复核模型、人工运营/客服

## 1. 判定目标

当群里收到新消息时，需要判断：

- 应该 `append_case` 到已有 Case
- 还是应该 `new_case` 新建 Case
- 还是只是 `ignore`（闲聊、补充语气词、无业务意义）

这一步的原则是：

**宁可多建 Case，也不要把不同问题误合并。**

## 2. 最核心原则

只有在“同业务对象 + 同问题表现 + 同一条排查链路”时，才允许合并。

如果下面任一项不同，优先新建 Case：

- 业务对象不同
- 上游/线路不同
- 问题表现不同
- 排查路径不同
- 影响模块不同

## 3. 什么叫“业务对象”

业务对象是指问题真正落在哪个系统或哪条链路上，而不是表面的症状词。

在云发单这套业务里，业务对象最好按四层理解：

- 第一层：上游
- 第二层：端别
- 第三层：登录设备
- 第四层：能力域

其中：

- 上游：易可云、探鲸、发单帝、云密码、小邪
- 端别：微信端、企微端
- 登录设备：Mac、iPad、AndroidPad、Windows、PC 等
- 能力域：登录、验证码、群消息接收、跟群、发图片、视频号、转链、订单等

只有当这几层里关键层级都一致时，才适合 `append_case`。

### 3.1 业务对象示例

- 悦拜 APP / 云发单前台
- office 后台 / 管理后台
- 探鲸
- 易可云
- 发单帝
- 云密码
- 小邪
- 微信端
- 企微端
- Mac / iPad / AndroidPad / Windows
- 转链服务
- 订单 / 统计
- 跟群 / 跟圈 / 群消息接收
- H5 页面 / DNS / 域名

### 3.2 一个更准确的结构示例

例如：

- 发单帝 -> 微信端 -> Mac -> 登录
- 探鲸 -> 微信端 -> iPad -> 登录
- 探鲸 -> 企微端 -> iPad -> 发图
- 小邪 -> 企微端 -> 回调消息 -> 图片下载
- 云密码 -> 微信端 -> Windows -> 群消息接收

这里的 `Mac / iPad / Windows` 是登录设备，不是和 `微信端 / 企微端` 同层的对象。

## 4. 允许合并的场景

以下更适合 `append_case`：

- 同一问题的补充描述
- 同一问题补充用户 ID、群 ID、wxid、手机号
- 同一问题补充截图、图片、视频、语音
- 同一问题追加“还有这个用户也出现了”
- 同一对象、同一症状、同一责任边界下的新增证据

### 4.1 例子：应该合并

Case 1：

- 探鲸账号无法发图片

后续补充：

- “这个用户也发不出去，wxid_xxx”
- “补一张报错截图”
- “同一台账号还是不行”

这类通常应合并。

## 5. 不允许合并的场景

以下更适合 `new_case`：

- 虽然都是“登录不上”，但系统对象不同
- 虽然都是“消息异常”，但一个是群消息接收，一个是跟群不生效
- 虽然都是“转链有问题”，但一个是淘宝主站口令，一个是三方短链解析
- 虽然同一个人反馈，但换了新业务对象或新模块
- 前一个 Case 已经是 A 系统，后一个消息落到 B 系统

### 5.1 明确不应合并的例子

#### 例 1

- “线上后台无法登录”
- “探鲸出现大面积封控，个微都无法登录”

都带“登录”，但业务对象不同：

- 前者更像后台/云发单系统问题
- 后者明确是探鲸/个微线路问题

应 `new_case`，不能合并。

#### 例 2

- “用户收不到验证码”
- “登录后一直转圈”

两个都在登录链路上，但症状和排查路径不同，优先拆分。

#### 例 3

- “office 后台看不到订单”
- “APP 页面空白”

一个是后台数据问题，一个是前台/H5 展示问题，应拆分。

#### 例 4

- “探鲸企微账号无法发图片”
- “探鲸企微无法跟发源头视频号”

同属探鲸企微，但问题能力不同：

- 一个是图片发送
- 一个是视频号兼容

优先拆分。

#### 例 5

- “发单帝微信端 Mac 登录失败”
- “发单帝微信端 Mac 登录需要验证码”

这两个更可能属于同一条排查链路，可以优先考虑合并。

#### 例 6

- “发单帝微信端 Mac 登录失败”
- “发单帝微信端 Mac 发图片失败”

虽然上游、端别、设备都一样，但能力域不同，优先拆分。

#### 例 7

- “探鲸微信端 iPad 登录失败”
- “探鲸企微端登录失败”

虽然同属探鲸，但端别不同，优先拆分。

## 6. 高风险误并信号

模型看到以下特征时，应默认谨慎，不要直接并案：

- 只因为出现相同动词：登录、失败、异常、卡顿、收不到
- 只因为同一个发送人连续发了多条
- 只因为优先级同为 P0/P1
- 只因为时间接近
- 只因为都属于“线上问题处理”

这些都不能单独作为 `append_case` 的依据。

## 7. 可以作为合并依据的强信号

- 明确提到“同一个问题”“还是刚才那个”
- 明确补充“这个用户也是同样报错”
- 补充的是同一 Case 的截图/视频/语音/账号信息
- 新消息与已有 Case 的业务对象、症状词、责任边界完全一致
- 最近消息就是对已有 Case 的延续，没有切换模块

## 8. 针对云发单业务的对象拆分建议

下面这些对象，默认视为不同 Case 维度：

- 云发单前台 / 悦拜 APP
- office 后台 / 管理后台
- 探鲸
- 易可云
- 发单帝
- 云密码
- 小邪
- 微信端
- 企微端
- Mac / iPad / AndroidPad / Windows
- 转链
- 订单
- 跟群 / 跟圈 / 订阅
- 页面/H5/DNS

如果两个问题跨了这些维度，优先新建。

更准确地说，建议优先按下面顺序判断边界：

1. 是否同一个上游
2. 是否同一个端别（微信端 / 企微端）
3. 是否同一个登录设备
4. 是否同一个能力域
5. 是否同一个具体症状与排查链路

只要前四步里有明显差异，就不要轻易 `append_case`。

## 9. 语义相近但应拆分的典型对照

### 9.1 登录类

- 云发单登录不上
- 探鲸登录不上
- 企微扫码登录失败
- 验证码无法登录

这几类不能默认并成一个“登录问题”。

### 9.2 消息类

- 群消息接收不到
- 跟群不生效
- 消息已下发但群里别人看不到
- 图片发不出去

都属于消息链路，但排查路线不同，优先拆分。

### 9.3 转链类

- 淘宝主站口令不能转
- 三方链接解析失败
- 某平台 CPS 链接丢失参数

不能简单并成“转链异常”，至少按平台/链路拆。

### 9.4 订单类

- 后台看不到新订单
- 订单同步延迟
- 出单统计不对

三者相近，但不一定是同一个根因，至少先保守拆分。

## 10. 给模型的执行建议

### 10.1 append_case 的门槛

只有同时满足时才允许：

- 明确是同上游
- 明确是同端别
- 明确是同登录设备（如果消息里有）
- 明确是同能力域
- 明确是同问题表现
- 明确是同一排查链路
- 新消息本质上只是补充证据

### 10.2 new_case 的默认条件

只要模型不能非常确定是同一个问题延续，就默认 `new_case`。

### 10.3 ignore 的适用情况

- 纯语气词：嗯？、还在吗、看看、收到
- 无业务信息的单独表情
- 无上下文的闲聊

但如果是图片、语音、wxid、群 ID、手机号等证据补充，不能直接 ignore，应结合前文判断。

## 11. 这份文档适合如何接到模型里

推荐用法：

- 放进 Case 分组模型的 system prompt
- 对 `append_case` 再做一次模型复核时，作为“边界说明”

不建议把这些规则写成大量硬编码 if/else。  
代码只负责流程校验；业务边界主要由模型参考这份说明来判断。
